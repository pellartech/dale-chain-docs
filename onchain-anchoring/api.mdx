`@dale/api` is the Fastify service that connects applications to DALE anchoring.
It prepares EIP-712 `TrustedForwarder` payloads, queues chain jobs, exposes job
status, lists indexed thread anchors, and serves generated OpenAPI.

<Card title="Recommended: use the SDK" icon="code" href="/onchain-anchoring/sdk">
  For most applications, use `@dale/sdk` instead of calling the anchoring API
  directly. The SDK handles payload normalization, thread helpers, EIP-712
  signing, submission, job polling, and event listing.
</Card>

## Authentication

Most routes require an API key:

```text
x-api-key: <DALE_API_KEY>
```

On devnet, any EVM account can prepare and submit thread jobs with a valid API
key. If another environment returns `account_not_allowed`, ask for that address
to be enabled before preparing or submitting.

Public routes are `GET /health`, `GET /openapi.json`, `GET /docs`, and
`POST /utils/hash`.

## Core routes

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/utils/hash` | Hash strings or canonicalized JSON with SHA-256. |
| `POST` | `/threads/prepare` | Prepare a signed thread creation or thread-admin payload. |
| `POST` | `/threads/submit` | Queue a signed thread creation or thread-admin chain job. |
| `POST` | `/threads/:threadId/anchors/prepare` | Prepare a signed thread-anchor payload. |
| `POST` | `/threads/:threadId/anchors/submit` | Queue a signed thread-anchor chain job. |
| `GET` | `/jobs/:id` | Read the current state of a queued chain job. |
| `GET` | `/threads/:threadId/anchors` | List indexed anchors for one thread. |

## Thread jobs

Thread creation and admin operations share `/threads/prepare` and
`/threads/submit`. The request body determines the operation.

Create a thread:

```json
{
  "owner": "0xf39fd6e51aad88f6f4ce6ab8827279cfffb92266",
  "threadSeed": "shipment-123"
}
```

The API normalizes the seed, computes the deterministic thread ID, reads the
owner's forwarder nonce, and returns an EIP-712 `ForwardRequest` payload to sign.

After signing, submit the same request with the returned signature fields:

```json
{
  "owner": "0xf39fd6e51aad88f6f4ce6ab8827279cfffb92266",
  "threadSeed": "shipment-123",
  "nonce": "7",
  "deadline": "1782130000",
  "signature": "0x..."
}
```

Thread-admin jobs use the same prepare/submit pair. Supported admin kinds are
participant grants, participant removals, handler updates, ownership transfer,
and ownership acceptance. The returned job is executed by the worker through
`TrustedForwarder.execute`, so the user remains the credited on-chain actor while
the backend pays gas.

See [Threads](/onchain-anchoring/threads) for the access model.

## Thread anchor request

Prepare an anchor for a thread:

```json
{
  "account": "0xf39fd6e51aad88f6f4ce6ab8827279cfffb92266",
  "type": "hashSha256",
  "data": "0x1111111111111111111111111111111111111111111111111111111111111111",
  "tag": "origin-next"
}
```

`type` can be `hashSha256`, `hashPoseidon`, `plaintext`, or
`thresholdEncrypted`.

| Type | `data` value |
| --- | --- |
| `hashSha256` | `0x`-prefixed 32-byte commitment. |
| `hashPoseidon` | `0x`-prefixed 32-byte commitment. |
| `plaintext` | UTF-8 string up to 1024 bytes. |
| `thresholdEncrypted` | `0x` ABI-encoded `ThresholdEnvelopeV1`. |

The API hashes optional `tag` values before they reach the contract. For
threshold-encrypted anchors, the API and contract treat `data` as ciphertext
envelope bytes; they do not decrypt it.

Signed submissions to `/threads/:threadId/anchors/submit` must include:

| Field | Description |
| --- | --- |
| `nonce` | Decimal uint256 string returned by prepare. |
| `deadline` | Unix timestamp returned by prepare. |
| `signature` | EVM signature for the returned `ForwardRequest`. |

Supplying only some signature fields is a validation error.

## Signed relay flow

1. Call `POST /threads/:threadId/anchors/prepare` with the anchor request.
2. Sign the returned EIP-712 payload with the credited account.
3. Call `POST /threads/:threadId/anchors/submit` with the same anchor request,
   the returned `nonce` and `deadline`, and the wallet `signature`.
4. The API verifies the signature before queueing the job.
5. The worker submits `TrustedForwarder.execute(...)`.
6. The contract resolves `_msgSender()` to the signer and stores the anchor only
   if that signer can write to the thread.

## Job status

Submit routes return a `jobId`. Track it with `GET /jobs/:id`.

| Status | Meaning |
| --- | --- |
| `queued` | The job is ready for the worker. |
| `processing` | The worker claimed the job and is submitting the transaction. |
| `submitted` | The transaction hash is known and indexing may still be pending. |
| `success` | The receipt emitted the expected event and indexed details are available. |
| `failed` | The worker hit a permanent error or exhausted retries. |

Successful status responses can include `transactionHash`, `eventId`,
`anchorId`, and `blockNumber`.

## Listing thread anchors

`GET /threads/:threadId/anchors` reads indexed rows from `anchor_events`.

Use these query parameters:

| Parameter | Description |
| --- | --- |
| `limit` | Integer from 1 to 100. Defaults to `50`. |
| `offset` | Integer greater than or equal to 0. Defaults to `0`. |
| `tag` | Optional exact tag filter. |

Hash anchors return an empty `data` value. Plaintext anchors return the indexed
UTF-8 payload. Threshold-encrypted anchors return the indexed envelope hex; the
API and indexer do not decrypt encrypted anchors.

## Error shape

Explicit API errors use this shape:

```json
{
  "error": {
    "code": "validation_error",
    "message": "Request validation failed",
    "details": [
      {
        "path": "data",
        "message": "Hash data must be a 0x-prefixed 32-byte hex string"
      }
    ]
  }
}
```

Common codes include `unauthorized`, `account_not_allowed`, `validation_error`,
`invalid_signature`, `signature_expired`, `contract_not_configured`, `not_found`,
and `internal_error`.
