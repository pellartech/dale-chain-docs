`CanonicalSupplyChain` is the on-chain proof layer for DALE supply-chain anchors.
It stores append-only anchor records inside deterministic supply-chain threads,
emits events for indexers, and delegates access checks to built-in thread policy
plus optional access handlers.

Applications should use the deployed proxy address. API services, workers,
indexers, SDK clients, and demos should point at the proxy rather than the
implementation contract.

## Contract modules

The contract package is split into focused Solidity files:

| Module | Purpose |
| --- | --- |
| `CanonicalSupplyChain.sol` | UUPS proof contract with anchor storage, validation, thread anchoring, cohort registry, and ERC-2771 support. |
| `Threads.sol` | Thread lifecycle, owner/participant access, handler hooks, and two-step ownership transfer. |
| `SupplyAnchor.sol` | Shared `AnchorType` enum and `SupplyAnchor` struct. |
| `IThreadAccessHandler.sol` | Optional handler interface for additive read/write policy. |
| `TrustedForwarder.sol` | OpenZeppelin `ERC2771Forwarder` with the `TrustedForwarder` EIP-712 domain name. |

## Anchor types

Each anchor has a thread ID, type, payload commitment, optional tag hash,
credited account, and global anchor ID. Anchor IDs are array indexes in the
append-only storage array.

| Contract value | API and SDK type | Behavior |
| --- | --- | --- |
| `0` | `hashSha256` | Stores a 32-byte SHA-256 commitment. |
| `1` | `hashPoseidon` | Stores a 32-byte Poseidon commitment. |
| `2` | `plaintext` | Stores plaintext bytes and records `sha256(data)`. |
| `3` | `thresholdEncrypted` | Stores an ABI-encoded threshold envelope and records its payload commitment. |

Plaintext payloads are capped at 1024 UTF-8 bytes. Hash payloads must be exactly
32 bytes.

## Stored shape

```solidity
struct SupplyAnchor {
    bytes32 threadId;
    bytes32 dataHash;
    bytes32 tagHash;
    address account;
    uint8 anchorType;
    bytes data;
}
```

Hash anchors store an empty `data` value and the supplied commitment in
`dataHash`. Plaintext anchors store bytes in `data` and a SHA-256 digest in
`dataHash`. Threshold-encrypted anchors store an ABI-encoded
`ThresholdEnvelopeV1` in `data`; `dataHash` stores the envelope
`payloadCommitment`.

## Event

```solidity
event SupplyAnchorCreated(
    bytes32 indexed threadId,
    bytes32 indexed dataHash,
    address indexed account,
    uint256 id,
    uint8 anchorType,
    bytes data,
    bytes32 tagHash
);
```

The indexer reads `SupplyAnchorCreated` logs and writes a queryable copy to the
application database. `threadId`, `dataHash`, and `account` are indexed on-chain
so callers can filter logs efficiently.

## Thread anchoring

`storeThreadAnchor(bytes32 threadId, uint8 anchorType, bytes data, bytes32 tagHash)`
stores an anchor credited to the effective sender and returns the new anchor ID.
The contract first builds and validates the anchor, then checks write access
against the thread before storing it.

Thread owners and participants have built-in read/write access. A thread can also
set an optional handler that grants additional thread-wide or per-anchor access.
Handlers are additive: they can grant access, but they cannot revoke owner or
participant access.

Important read helpers:

| Function | Purpose |
| --- | --- |
| `canReadAll(bytes32 threadId, address account)` | Checks thread-wide read access. |
| `canRead(bytes32 threadId, uint256 anchorId, address account)` | Checks thread-wide read access, then per-anchor handler access. |
| `canWrite(bytes32 threadId, uint8 anchorType, bytes data, bytes32 tagHash, address account)` | Checks whether an account may write the next anchor. |
| `anchors(uint256 id)` | Reads a stored anchor by global anchor ID. |
| `anchorCount()` | Returns `anchors.length`. |

See [Threads](/onchain-anchoring/threads) for the thread lifecycle and access
model.

## Gas-sponsored calls

There are no separate per-function relayed variants. Gas-sponsored actions use
`TrustedForwarder.execute` with a deadline-bound EIP-712 `ForwardRequest`.

The signed primary type is:

```text
ForwardRequest(address from,address to,uint256 value,uint256 gas,uint256 nonce,uint48 deadline,bytes data)
```

The `data` field is the ABI-encoded calldata for the canonical contract function,
such as `createThread`, `addThreadParticipant`, or `storeThreadAnchor`.

| EIP-712 field | Value |
| --- | --- |
| `domain.name` | `TrustedForwarder` |
| `domain.version` | `1` |
| `domain.chainId` | `16001` on `devnet-1` |
| `domain.verifyingContract` | Trusted forwarder address |
| `message.to` | `CanonicalSupplyChain` proxy address |
| `message.nonce` | `TrustedForwarder.nonces(from)` |
| `message.deadline` | Unix timestamp after which the signature is invalid |
| `message.data` | Encoded inner contract call |

The forwarder verifies the signer, nonce, and deadline, then calls the contract
with the signer appended to calldata. Inside `CanonicalSupplyChain`, `_msgSender()`
resolves to the original signer.

## Decryption cohort registry

Threshold-encrypted anchors use an owner-managed registry:

| Function | Purpose |
| --- | --- |
| `registerDecryptionCohort(...)` | Registers a threshold keyset, suite ID, threshold, node count, activation, expiry, and active flag. |
| `setActiveDecryptionCohort(uint32 cohortId)` | Selects the active cohort for new threshold-encrypted anchors. |
| `setDecryptionNode(...)` | Records a node index, active flag, public share, signing address, and endpoint. |
| `getDecryptionCohort(uint32 cohortId)` | Reads public cohort metadata. |
| `getDecryptionNode(uint32 cohortId, uint16 nodeIndex)` | Reads public node metadata. |

Publishers use the active cohort to encrypt new `thresholdEncrypted` anchors.
Requesters and decryption nodes use the cohort and node metadata to authenticate
decrypt-share responses.

## Validation

The contract rejects invalid anchor types, hash payloads with the wrong byte
length, oversized plaintext payloads, threshold envelopes with invalid context,
unauthorized thread writes, expired or invalid forwarder requests, and stale
forwarder nonces.
