`@dale/sdk` is the TypeScript client for DALE anchoring. It wraps API calls, viem
contract reads and writes, anchor normalization, signed `TrustedForwarder`
payloads, event parsing, polling, and threshold-encrypted anchor helpers.

The SDK expects callers to provide viem clients. Use a `publicClient` for reads,
watches, and transaction waits. Use a `walletClient` for signing and writes.

## Create a client

```ts
import { createDaleClient } from "@dale/sdk";

const dale = createDaleClient({
  apiBaseUrl: "http://dale-devnet-alb-305857230.ap-southeast-1.elb.amazonaws.com",
  apiKey: "<DALE_API_KEY>",
  contractAddress: "0xBCb9211802e530Fd6BE7Cd9157715249a15cAD9c",
  defaultAccount,
  publicClient,
  walletClient,
});
```

| Option | Purpose |
| --- | --- |
| `apiBaseUrl` | Base URL for `@dale/api`. |
| `apiKey` | Value sent as `x-api-key`. |
| `contractAddress` | `CanonicalSupplyChain` proxy address. |
| `defaultAccount` | Optional default viem `Account` or address. |
| `publicClient` | Required for reads, watches, threshold metadata, and transaction waits. |
| `walletClient` | Required for signing and writes. |
| `fetch` | Optional custom fetch implementation. |
| `pollIntervalMs` | Optional default polling delay. |
| `timeoutMs` | Optional transaction wait timeout. |

When a browser creates a local private key, pass a viem `Account` object rather
than only an address. That lets viem sign locally.

## Create a thread

Anchors live inside deterministic supply-chain threads. Create a thread before
submitting shipment or status anchors:

```ts
const result = await dale.submitThreadViaApi({
  owner: defaultAccount,
  threadSeed: "shipment-123",
});

console.log(result.jobId, result.prepared.threadId);
```

`submitThreadViaApi` prepares a `TrustedForwarder` payload, signs it locally, and
queues a chain job through the API. Use `createThreadDirect` when the connected
wallet should submit the contract transaction itself.

## Anchor inputs

You can submit an explicit anchor:

```ts
await dale.submitThreadAnchorViaApi({
  account: defaultAccount,
  threadId,
  anchor: {
    type: "hashSha256",
    data: "0x1111111111111111111111111111111111111111111111111111111111111111",
    tag: "shipment-123",
  },
});
```

You can also submit JSON:

```ts
await dale.submitThreadAnchorViaApi({
  account: defaultAccount,
  threadId,
  anchor: {
    json: {
      shipmentId: "shipment-123",
      status: "packed",
    },
    tag: "shipment-123",
    visibility: "opaque",
  },
});
```

| Visibility | Resulting anchor type | Behavior |
| --- | --- | --- |
| `opaque` or omitted | `hashSha256` | Canonical JSON is SHA-256 hashed and only the digest is anchored. |
| `transparent` | `plaintext` | Canonical JSON is stored on-chain as plaintext. |

SDK JSON `visibility: "opaque"` produces a hash commitment. It does not produce
a threshold envelope. Use the dedicated threshold helpers for threshold
encryption.

The SDK canonicalizes JSON by sorting object keys. It rejects non-finite numbers
and a bare top-level `undefined`.

## Submission modes

| Method | Path | Who pays gas | On-chain credited account |
| --- | --- | --- | --- |
| `submitThreadAnchorDirect` | Direct contract call to `storeThreadAnchor` | Connected wallet | Transaction signer |
| `submitThreadAnchorViaApi` | API prepare, EIP-712 sign, API submit, worker relay | Worker relayer | Signed account |

Use `submitThreadAnchorDirect` when the connected wallet should submit the
transaction itself:

```ts
const result = await dale.submitThreadAnchorDirect({
  account: defaultAccount,
  threadId,
  anchor: {
    json: {
      shipmentId: "shipment-123",
      status: "packed",
    },
    visibility: "transparent",
  },
  wait: true,
});

console.log(result.transactionHash, result.event);
```

Use `submitThreadAnchorViaApi` when the user signs locally and the backend
relays:

```ts
const result = await dale.submitThreadAnchorViaApi({
  account: defaultAccount,
  threadId,
  anchor: {
    type: "plaintext",
    data: "shipment 123 packed at dock 4",
    tag: "shipment-123",
  },
});

console.log(result.jobId, result.prepared.dataHash);
```

Track the returned `jobId` with `getJobStatus`.

## Thread admin

Use `submitThreadAdminViaApi` for participant and handler changes:

```ts
await dale.submitThreadAdminViaApi({
  kind: "addParticipant",
  account: threadOwner,
  threadId,
  participant: portOperator,
});
```

Thread admin calls use the same gas-sponsored `TrustedForwarder` flow as anchor
submissions.

## Threshold-encrypted anchors

Threshold-encrypted anchors are a local proof of concept. They encrypt payloads
before publication, store an envelope on-chain, and allow authorized requesters
to recover plaintext by collecting enough decryption-node shares. Do not use
this mode for real protected data until the mock threshold suite is replaced
with reviewed cryptography.

Submit an encrypted anchor through the API:

```ts
const result = await dale.submitThresholdEncryptedAnchorViaApi({
  account: defaultAccount,
  threadId,
  json: {
    shipmentId: "shipment-123",
    temperature: "2C",
  },
  tag: "shipment-123-private",
});

console.log(result.jobId, result.encrypted.expectedAnchorId);
```

Decrypt an indexed threshold-encrypted anchor:

```ts
const decrypted = await dale.decryptThresholdAnchor({
  anchorId: 42n,
  requester: defaultAccount,
});

console.log(decrypted.text);
```

The SDK fetches the stored envelope, verifies authenticated data, builds a
short-lived EIP-712 decrypt request, contacts active cohort nodes at
`/v1/decrypt-share`, verifies share responses, combines enough valid shares, and
decrypts locally.

## API helpers

| Method | Purpose |
| --- | --- |
| `prepareThread` | Calls `/threads/prepare` for thread creation. |
| `prepareThreadAdmin` | Calls `/threads/prepare` for thread admin jobs. |
| `prepareThreadAnchor` | Calls `/threads/:threadId/anchors/prepare`. |
| `getJobStatus(jobId)` | Calls `/jobs/:id`. |
| `listThreadEvents(options)` | Calls `/threads/:threadId/anchors`. |
| `hashPayload(data)` | Calls `/utils/hash` for SHA-256 hashing. |

`listThreadEvents` accepts `threadId`, `limit`, `offset`, and `tag`.

## Reads and watchers

| Method | Purpose |
| --- | --- |
| `getSupplyAnchor(id)` | Reads `anchors(id)` from the contract. |
| `getAnchorCount()` | Reads `anchorCount()`. |
| `getThread(threadId)` | Reads thread metadata. |
| `canReadAll(threadId, account)` | Checks thread-wide read access. |
| `canRead(threadId, anchorId, account)` | Checks anchor-specific read access. |
| `getActiveDecryptionCohortId()` | Reads the active threshold cohort ID. |
| `getDecryptionCohort(cohortId)` | Reads public cohort metadata. |
| `getDecryptionNode(cohortId, nodeIndex)` | Reads public node metadata. |
| `watchContractEvents(options)` | Uses viem to watch live `SupplyAnchorCreated` logs. |
| `watchIndexedEvents(options)` | Polls the API event list and calls `onEvent` for unseen indexed events. |

`watchContractEvents` can filter by `account`, `dataHash`, and `tagHash`.
`watchIndexedEvents` accepts `threadId`, `tag`, `onError`, `pollIntervalMs`, and
`signal`.

## Errors

API errors throw `DaleApiError` with `status`, `code`, `message`, and optional
`details`. Local validation and missing-client cases throw regular `Error`
values.
